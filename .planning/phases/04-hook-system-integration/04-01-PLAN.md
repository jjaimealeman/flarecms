<objective>
Extend the hook system foundation so route handlers can fire before/after lifecycle hooks with cancellation support, create a hooks singleton, and thread the Hono app reference through to PluginContext so plugins receive it in activate().

Purpose: The existing HookSystemImpl is fully functional but (a) HOOKS constants lack before/after variants and content:unpublish, (b) execute() doesn't propagate cancellation to callers, (c) there's no importable singleton for route files, and (d) PluginContext never receives the app reference — plugins can't register middleware/routes during activate(). This plan fixes all four gaps so Plan 02 can wire hooks into routes.

Output: Extended types, updated hook-system, hooks singleton module, app reference threaded through bootstrap -> PluginManager -> PluginContext.
</objective>

**Create plugins/hooks-singleton.ts:**

A module-level singleton pattern (same pattern as services/cache.ts kvInitialized). Workers isolates persist module state across requests within the same cold start window.

```typescript
import { HookSystemImpl } from './hook-system'

// Module-level singleton — persists per Workers isolate
let hookSystemInstance: HookSystemImpl | null = null

export function getHookSystem(): HookSystemImpl {
  if (!hookSystemInstance) {
    hookSystemInstance = new HookSystemImpl()
  }
  return hookSystemInstance
}

// For testing: reset singleton
export function resetHookSystem(): void {
  if (hookSystemInstance) {
    hookSystemInstance.clear()
  }
  hookSystemInstance = null
}
```

Add re-export: `export { getHookSystem, resetHookSystem } from './hooks-singleton'`

**Thread app reference through to PluginContext — this is the critical wiring for SC #3:**

The chain is: `createSonicJSApp()` creates `app` -> passes to `bootstrapMiddleware(config, app)` -> PluginBootstrapService gets app -> PluginManager.initialize() receives PluginContext with `app` -> PluginManager stores `this.context` (which now includes `app`) -> when `install()` or `loadPlugins()` constructs a `pluginContext` at lines ~129 and ~194, it spreads `this.context` which includes `app` -> plugin's `activate()` receives PluginContext with `app`.

Concrete changes:

1. **In app.ts** (`createSonicJSApp()`): Change `bootstrapMiddleware(config)` call to `bootstrapMiddleware(config, app)`. Import getHookSystem from `./plugins/hooks-singleton` for singleton availability.

2. **In middleware/bootstrap.ts** (`bootstrapMiddleware()`): Accept second parameter `app?: Hono`. The bootstrap function currently creates a `PluginBootstrapService` — it does NOT currently create a `PluginManager` or call `PluginManager.initialize()`. The PluginBootstrapService handles core plugin DB records, not the PluginManager lifecycle. So instead, store the app reference in a module-level variable that PluginManager can access. Add:

```typescript
import { Hono } from 'hono'

let appReference: Hono | undefined

export function getAppReference(): Hono | undefined {
  return appReference
}

export function bootstrapMiddleware(config: SonicJSConfig = {}, app?: Hono) {
  appReference = app
  return async (c: Context<{ Bindings: Bindings }>, next: Next) => {
    // ... existing bootstrap logic unchanged ...
  }
}
```

3. **In plugins/plugin-manager.ts**: Import `getAppReference` from `../middleware/bootstrap`. At both PluginContext construction sites (lines ~129 and ~194), the code does `...this.context` spread. Since `this.context` is set in `initialize()`, ensure `initialize()` includes the app reference. Update the `initialize()` method to pull the app reference:

```typescript
async initialize(context: PluginContext): Promise<void> {
  console.info('Initializing plugin system...')

  // Ensure app reference is available in context
  if (!context.app) {
    const { getAppReference } = await import('../middleware/bootstrap')
    const appRef = getAppReference()
    if (appRef) {
      context.app = appRef
    }
  }

  this.context = context
  // ... rest unchanged
}
```

This way, when `install()` at line 129 does `...this.context`, the spread includes `app`. Same for `uninstall()` at line 194.

The result: any plugin's `activate(context)` callback receives `context.app` set to the Hono app instance, satisfying SC #3.
