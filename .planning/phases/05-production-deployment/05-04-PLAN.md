  <action>
    The Astro API client (`src/lib/sonicjs.ts`) already reads `import.meta.env.SONICJS_API_URL` with `http://localhost:8787` fallback. This is correct for local dev. For production on Cloudflare Pages, the env var must be set.

    **Step 1: Add SONICJS_API_URL to wrangler.jsonc**

    Read `my-astro-site/wrangler.jsonc` and add a `vars` section (if not already present) with the production CMS URL:

    ```jsonc
    {
      // ...existing config...
      "vars": {
        "SONICJS_API_URL": "https://my-sonicjs-app-production.workers.dev"
      }
    }
    ```

    NOTE: The actual production URL should match what was deployed in Plan 03. Read the 05-03-SUMMARY.md to get the exact URL. If the summary contains a different URL (e.g., custom domain), use that instead.

    **Step 2: Verify sonicjs.ts handles the env var correctly**

    Read `my-astro-site/src/lib/sonicjs.ts`. The current code is:
    ```typescript
    const API_URL = import.meta.env.SONICJS_API_URL || "http://localhost:8787";
    ```

    For Cloudflare Pages with `@astrojs/cloudflare` adapter, `import.meta.env` reads from wrangler.jsonc vars at build time AND from Cloudflare environment at runtime. The `vars` in wrangler.jsonc will be available.

    However, there is a subtlety: Astro on Cloudflare Pages uses SSR — `import.meta.env` may need to be a server-side env var. Check if the variable needs `PUBLIC_` prefix for client-side or if all API calls are server-side (they appear to be server-side based on the SSR setup). Since all fetch calls are in server-side `.astro` pages, the var does NOT need `PUBLIC_` prefix.

    If the env var pattern looks correct as-is, no changes needed to sonicjs.ts. If there are issues with how Cloudflare Pages exposes vars, the fallback pattern handles it.

    **Step 3: Verify Astro build works**

    Run `cd my-astro-site && npm run build` to confirm the Astro site builds without errors. Do NOT run the dev server.
  </action>

    3. `ls my-astro-site/dist/` shows build output
  </verify>
