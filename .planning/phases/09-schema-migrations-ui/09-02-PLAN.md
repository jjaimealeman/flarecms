Modify the three field CRUD routes in `admin-collections.ts` to record schema migrations:

**Import:** Add `import { SchemaMigrationService } from '../services/schema-migration'` at the top of the file.

**1. POST /:id/fields (Add field) -- around line 616:**

Before the schema update logic (before `schema.properties[fieldName] = fieldConfig`):
- Capture `const previousSchema = JSON.parse(JSON.stringify(schema))` (deep clone)
- Add validation: `const validation = SchemaMigrationService.validateFieldChange({ managed: collection.managed === 1, fieldName, existingFields: Object.keys(schema.properties || {}) })`
- If `!validation.valid`, return `c.json({ success: false, error: validation.error })`

After the schema is successfully updated in D1 (after the `updateSchemaStmt.bind(...).run()` succeeds):
- Create migration service: `const migrationService = new SchemaMigrationService(db)`
- Record: `await migrationService.recordMigration({ collectionId, collectionName: collection.display_name || collection.name, changes: [{ type: 'add_field', fieldName, fieldLabel, fieldType, newConfig: fieldConfig }], previousSchema, userId: user?.userId || null })`
- Get the user from `c.get('user')` (already available in the route since requireAuth is applied)

**2. PUT /:collectionId/fields/:fieldId (Edit field) -- around line 767:**

For schema fields (the `if (fieldId.startsWith('schema-'))` branch):
- Before modifying: capture `const previousSchema = JSON.parse(JSON.stringify(schema))`
- Capture the previous field config: `const previousConfig = { ...schema.properties[fieldName] }`
- After successful D1 update:
  ```typescript
  const migrationService = new SchemaMigrationService(db)
  await migrationService.recordMigration({
    collectionId,
    collectionName: collection.display_name || collection.name,
    changes: [{ type: 'modify_field', fieldName, fieldLabel, fieldType, previousConfig, newConfig: updatedFieldConfig }],
    previousSchema,
    userId: c.get('user')?.userId || null
  })
  ```

For legacy content_fields fields: skip migration tracking (these are deprecated).

**3. DELETE /:collectionId/fields/:fieldId (Delete field) -- around line 925:**

For schema fields (the `if (fieldId.startsWith('schema-'))` branch):
- Before deleting: capture `const previousSchema = JSON.parse(JSON.stringify(schema))`
- Capture the removed config: `const removedConfig = { ...schema.properties[fieldName] }`
- After successful D1 update:
  ```typescript
  const migrationService = new SchemaMigrationService(db)
  await migrationService.recordMigration({
    collectionId,
    collectionName: collection.display_name || collection.name,
    changes: [{ type: 'remove_field', fieldName, fieldLabel: removedConfig.title || fieldName, fieldType: removedConfig.type, previousConfig: removedConfig }],
    previousSchema,
    userId: c.get('user')?.userId || null
  })
  ```

**Error handling:** Wrap each `recordMigration` call in try/catch. If recording fails, log the error but do NOT fail the field operation -- the schema change itself already succeeded. Migration tracking is best-effort.

**Managed collection guard:** The `validateFieldChange` call in the add route should also be added to the edit and delete routes. Check `collection.managed === 1` and return an error if true.
