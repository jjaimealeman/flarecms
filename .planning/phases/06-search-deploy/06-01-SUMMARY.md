# Phase 6 Plan 1: Search Index Endpoint Summary

**MiniSearch installed with shared config and SSR search index endpoint serving serialized index of all CMS content with heading-level granularity**

## What Was Done

### Task 1: Install MiniSearch and create shared search config
- Installed `minisearch@7.2.0` in the site package
- Created `search-config.ts` with shared MiniSearch options, SearchDocument type, and stripMarkdown utility
- Exports: `MINISEARCH_OPTIONS`, `SEARCH_FIELDS`, `SEARCH_STORE_FIELDS`, `SearchDocument`, `stripMarkdown`
- `stripMarkdown` strips headings, bold, italic, code blocks, inline code, links, images, blockquotes, list markers, and collapses whitespace
- Commit: `d7ac383`

### Task 2: Create server-side search index endpoint
- Created `/api/search-index.json` SSR endpoint that builds and serializes a MiniSearch index
- Fetches docs sections, docs pages, blog posts, and news articles in parallel
- Creates main document per page plus sub-documents for each h2/h3 heading with anchor IDs
- Heading IDs generated by lowercasing and replacing non-word chars with hyphens
- Blog posts indexed with section "Blog", news with section "News"
- Response headers: `Content-Type: application/json`, `Cache-Control: public, max-age=300, s-maxage=300`
- Error handling returns 500 with JSON error response
- Commit: `a207419`

## Decisions Made

| Decision | Rationale |
|----------|-----------|
| Shared search-config.ts for server + client | MiniSearch.loadJSON() requires identical options at both ends |
| All CMS content indexed (docs, blog, news) | Unified search experience across all content types |
| h2/h3 sub-documents with heading anchor IDs | Deep-link search results to specific page sections |
| Regex-based markdown stripping (not full render) | Sufficient for trusted CMS content; avoids expensive Shiki rendering |
| 300s Cache-Control on index response | Balance freshness with CDN performance |
