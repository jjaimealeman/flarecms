  <action>
  Add bracket-syntax filter parsing to the `parseFromQuery()` static method. Insert the new parsing block AFTER the existing `simpleFieldMappings` loop (around line 401) and BEFORE the limit/offset/sort parsing.

  1. Add an `OPERATOR_MAP` constant mapping short and long operator names to `FilterOperator` types:
     - 'equals'/'eq' -> 'equals'
     - 'not_equals'/'ne' -> 'not_equals'
     - 'like' -> 'like'
     - 'contains' -> 'contains'
     - 'gt' -> 'greater_than', 'gte' -> 'greater_than_equal'
     - 'lt' -> 'less_than', 'lte' -> 'less_than_equal'
     - 'in' -> 'in', 'not_in' -> 'not_in'
     - 'exists' -> 'exists'

  2. Iterate `Object.entries(query)` and match keys against `/^filter\[([^\]]+)\]\[([^\]]+)\]$/`. For each match:
     - Extract `field` (match[1]) and look up `op` in OPERATOR_MAP (match[2].toLowerCase())
     - If valid operator and field, push `{ field, operator: op, value }` to `filter.where.and`
     - Skip keys that don't match the regex (they're handled by existing code)

  3. Also add support for the `-field` sort shorthand (e.g., `?sort=-created_at` for descending). Currently sort only parses JSON. Add a block: if `query.sort` is a string that does NOT start with `[` (not JSON), treat it as a field name. If prefixed with `-`, set order to 'desc', otherwise 'asc'. Wrap in `[{ field, order }]` array.

  The existing `sanitizeFieldName()` method already prevents SQL injection. No additional sanitization needed.

  Do NOT modify the existing `?where=<JSON>`, `?status=`, `?collection_id=` parsing — those must continue working.
