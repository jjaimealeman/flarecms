  **Part B: Create API token service**

  Create `sonicjs-fork/packages/core/src/services/api-tokens.ts`:

  1. `hashToken(token: string): Promise<string>` — SHA-256 using `crypto.subtle.digest()`, returns hex string

  2. `createApiToken(db: D1Database, params: { name: string, userId: string, allowedCollections: string[] | null, expiresAt: number | null }): Promise<{ tokenValue: string, tokenPrefix: string, id: string }>`
     - Generate token: `st_` + 24 random bytes as hex (48 hex chars)
     - `tokenPrefix` = first 10 chars of tokenValue (e.g., `st_abc1234`)
     - Hash token with `hashToken()`
     - INSERT into api_tokens: id (UUID), name, token (hash), token_hash (hash), token_prefix, user_id, permissions ('["content:read"]'), allowed_collections (JSON or null), is_read_only (1), expires_at, created_at
     - Return `{ tokenValue, tokenPrefix, id }` — tokenValue is shown ONCE

  3. `validateApiToken(db: D1Database, token: string): Promise<{ valid: boolean, tokenRecord?: any, error?: string }>`
     - Hash the incoming token
     - SELECT from api_tokens WHERE token_hash = ? AND (expires_at IS NULL OR expires_at > Date.now())
     - If found: update last_used_at, return `{ valid: true, tokenRecord }`
     - If not found: return `{ valid: false, error: 'Invalid or expired API token' }`

  4. `revokeApiToken(db: D1Database, tokenId: string): Promise<void>` — DELETE from api_tokens

  5. `listApiTokens(db: D1Database, userId?: string): Promise<Array<any>>` — SELECT id, name, token_prefix, allowed_collections, is_read_only, expires_at, last_used_at, created_at (never return hash)

  Export all. Add to core `index.ts`.

  **Part A: Add X-API-Key support to auth middleware (auth.ts)**

  Modify `requireAuth()` (or create a new middleware that runs before it) to check for `X-API-Key` header BEFORE JWT:

  1. Get `const apiKey = c.req.header('X-API-Key')`
  2. If apiKey present:
     - Import and call `validateApiToken(db, apiKey)` from api-tokens service
     - If invalid: return 401
     - If valid: set context variables:
       - `c.set('user', { userId: tokenRecord.user_id, email: '', role: 'viewer' })`
       - `c.set('apiToken', tokenRecord)` — includes `allowed_collections`, `is_read_only`
     - Call `next()`
  3. If no apiKey, fall through to existing JWT auth logic

  **Part B: Enforce collection scoping on GET routes (api.ts)**

  In the GET `/api/collections/:collection/content` route (in api.ts), add a check:
  - If `c.get('apiToken')` exists:
    - Parse `allowed_collections` (JSON array or null)
    - If not null and the requested collection ID is not in the array, return 403
    - If `is_read_only` is 1, ensure this is a GET request (it already is, but document the enforcement)

  In the content CRUD routes (POST, PUT, DELETE):
  - If `c.get('apiToken')` exists and `is_read_only` is 1, return `c.json({ error: 'API token is read-only' }, 403)`

  **Part C: Admin UI for token management**

  Create `sonicjs-fork/packages/core/src/routes/admin-api-tokens.ts`:

  1. GET `/admin/api-tokens` — List all tokens (showing prefix, name, allowed collections, expiry, last used). Only admin role.
  2. GET `/admin/api-tokens/create` — Form to create token (name, collection checkboxes, optional expiry date). Only admin role.
  3. POST `/admin/api-tokens` — Creates token, shows the full token value ONCE in a success page with a copy button and warning "This token will not be shown again."
  4. POST `/admin/api-tokens/:id/revoke` — Revokes (deletes) token. Only admin role.

  Register the routes in the admin router. Use the existing admin template patterns (HTMX, server-rendered HTML).
