  <files>sonicjs-fork/packages/core/src/lib/logger.ts</files>
  <action>
    Create `sonicjs-fork/packages/core/src/lib/logger.ts` with a lightweight structured logger.

    The logger wraps console.log/warn/error with consistent JSON field structure. Workers Logs automatically indexes JSON fields from console output — no library needed.

    Implementation:
    ```typescript
    type LogLevel = 'info' | 'warn' | 'error'

    export const log = {
      info(event: string, fields: Record<string, unknown> = {}): void {
        console.log(JSON.stringify({ level: 'info', event, ts: Date.now(), ...fields }))
      },
      warn(event: string, fields: Record<string, unknown> = {}): void {
        console.warn(JSON.stringify({ level: 'warn', event, ts: Date.now(), ...fields }))
      },
      error(event: string, fields: Record<string, unknown> = {}): void {
        console.error(JSON.stringify({ level: 'error', event, ts: Date.now(), ...fields }))
      },
    }
    ```

    Use JSON.stringify to ensure Workers Logs gets clean parseable JSON (not `[Object object]`). Include `ts` (timestamp as epoch ms) in every log entry for correlation.

    Also ensure this is exported from the package index if there's a barrel export file — check `sonicjs-fork/packages/core/src/index.ts` for re-exports.
  </action>
  <verify>File exists at the path. TypeScript compiles without errors: `cd sonicjs-fork && npx tsc --noEmit --pretty 2>&1 | grep -i logger || echo "No logger errors"`</verify>
  <done>logger.ts exists with log.info, log.warn, log.error functions that produce structured JSON via console methods</done>

    **Auth routes (auth.ts):**
    - After successful login (JWT generated): `log.info('auth.login', { userId, email: normalizedEmail, ip: c.req.header('CF-Connecting-IP') })`
    - After failed login (wrong password or user not found): `log.warn('auth.login_failed', { email, reason: 'invalid_password' OR 'user_not_found', ip: c.req.header('CF-Connecting-IP') })`
    - After successful register: `log.info('auth.register', { userId, email: normalizedEmail })`
    - On logout: `log.info('auth.logout', { userId })` (if logout route exists and has user context)

    **Auth middleware (auth.ts in middleware/):**
    - On permission denied (403 responses): `log.warn('auth.permission_denied', { userId, resource: c.req.path, method: c.req.method })`

    **Content routes (api.ts):**
    Find the POST (create), PUT (update), and DELETE handlers for content. Add:
    - After successful create: `log.info('content.created', { contentId, collectionId, userId })`
    - After successful update: `log.info('content.updated', { contentId, collectionId, userId })`
    - After successful delete: `log.info('content.deleted', { contentId, userId })`
    - After publish (status change to published): `log.info('content.published', { contentId, collectionId, userId })`
    - After unpublish: `log.info('content.unpublished', { contentId, collectionId, userId })`

    IMPORTANT: All log calls must be non-blocking. They should NOT be in try/catch that would swallow errors from the main logic. Place them AFTER the successful database operation, before the response is returned. If a log call itself throws (unlikely with console), it should not break the response — but console methods don't throw, so this is informational.
