<objective>
Create the Draft Content API endpoints that store and retrieve preview draft data in Cloudflare KV.

Purpose: This is the foundation for live preview -- the CMS editor will POST draft form data here, and the Astro preview route will GET it via token. Using KV with TTL keeps drafts ephemeral (5 min) and the deterministic key pattern (preview:{userId}:{collectionId}:{contentId}) prevents race conditions from rapid typing.

Output: Two API endpoints mounted at /api/preview/draft (POST) and /api/preview/draft/:token (GET), plus route registration in core.
</objective>

<task type="auto">
  <name>Task 1: Create Draft API route file</name>
  <files>packages/core/src/routes/admin-preview.ts</files>
  <action>
Create a new Hono route file `admin-preview.ts` with two endpoints:

1. `POST /draft` -- Store draft data in KV:
   - Apply `requireAuth()` middleware to all routes (same pattern as admin-content.ts line 183)
   - Accept JSON body: `{ collectionId: string, contentId?: string, data: Record<string, any>, title?: string, slug?: string, status?: string }`
   - Get user from `c.get('user')` (set by requireAuth)
   - Build deterministic KV key: `preview:${user.userId}:${collectionId}:${contentId || 'new'}`
   - Store in KV via `c.env.CACHE_KV.put(key, JSON.stringify(draft), { expirationTtl: 300 })`
   - The draft object should include: collectionId, contentId, title, slug, status (default 'draft'), data, userId, createdAt (Date.now())
   - Return JSON: `{ token: previewKey, expiresIn: 300 }`

2. `GET /draft/:token` -- Retrieve draft data:
   - The token param is the full KV key (URL-encoded)
   - Fetch from KV via `c.env.CACHE_KV.get(token)`
   - If null, return 404: `{ error: 'Preview expired or not found' }`
   - If found, return 200: `{ data: JSON.parse(raw) }`

Use the same Hono generic types as other admin routes: `Hono<{ Bindings: Bindings; Variables: Variables }>`. Import Bindings and Variables from the same location as admin-content.ts does.

NOTE: The GET endpoint for draft retrieval should NOT require auth -- it uses the token itself as authentication (the token is opaque and short-lived). Only the POST endpoint needs requireAuth. This is important because the Astro site (different origin) will call GET to fetch draft data and it does not have CMS JWT cookies.

Actually, apply requireAuth to all routes BUT make the GET /draft/:token endpoint use a separate sub-route without auth. The simplest pattern: create two Hono instances -- one with auth for POST, one without for GET -- and merge them. Or use a single instance and apply requireAuth selectively to POST only.
  </action>
