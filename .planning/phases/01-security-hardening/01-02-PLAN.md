    SEC-01 — JWT_SECRET startup assertion:
    The .dev.vars has a proper JWT_SECRET, so the server should be running normally.
    Verify the assertion is active by confirming the server responds to requests (meaning JWT_SECRET passed validation):
    ```bash
    curl -s -o /dev/null -w "%{http_code}" http://localhost:8787/api/health
    ```
    Expected: 200 (server is running, JWT_SECRET assertion passed).
    If the server is NOT running (connection refused), this is itself evidence the assertion may be blocking — investigate.

    SEC-02 — PBKDF2 password hashing:
    Runtime verification: Log in with admin credentials via curl:
    ```bash
    curl -s -X POST http://localhost:8787/auth/login \
      -H "Content-Type: application/json" \
      -d '{"email":"admin@example.com","password":"admin123"}' | head -c 500
    ```
    Then check if the admin user's password_hash was migrated:
    ```bash
    cd /home/jaime/www/_github/sonicjs/my-astro-cms && npx wrangler d1 execute DB --local --command "SELECT id, substr(password_hash, 1, 30) as hash_prefix FROM users WHERE email='admin@example.com'" 2>/dev/null
    ```
    The hash MUST now start with "pbkdf2:" if migration worked. If it still shows a raw SHA-256 hash, SEC-02 has failed.

    SEC-03 — Rate limiting:
    ```bash
    # Check rate limit headers on a normal response
    curl -s -D - http://localhost:8787/auth/login \
      -X POST \
      -H "Content-Type: application/json" \
      -d '{"email":"test@test.com","password":"wrong"}' 2>&1 | grep -i "x-ratelimit\|retry-after"
    ```
    X-RateLimit headers MUST appear on auth responses. If they don't, CACHE_KV binding may not be configured correctly — check wrangler.toml.

    SEC-04 — Security headers:
    ```bash
    curl -s -D - http://localhost:8787/api/health 2>&1 | grep -i "x-content-type-options\|x-frame-options\|strict-transport\|referrer-policy\|permissions-policy"
    ```
    Expected: X-Content-Type-Options: nosniff and X-Frame-Options: SAMEORIGIN MUST be present.
    Note: HSTS (Strict-Transport-Security) may be skipped when ENVIRONMENT=development — this is correct behavior.

    Also verify headers on a non-API route:
    ```bash
    curl -s -D - http://localhost:8787/admin 2>&1 | grep -i "x-content-type-options\|x-frame-options"
    ```
    Security headers should be present on ALL routes, not just /api/*.

    SEC-05 — CORS (scope: /api/* routes):
    NOTE: CORS middleware in the fork is applied to /api/* routes only. Admin routes (/admin/*) are not CORS-protected because they are same-origin browser requests, not cross-origin API calls. This is intentional — CORS protects the API surface.
    ```bash
    # Test allowed origin on API route
    curl -s -D - -H "Origin: http://localhost:4321" http://localhost:8787/api/collections 2>&1 | grep -i "access-control-allow-origin"

    # Test disallowed origin on API route (should NOT return Access-Control-Allow-Origin)
    curl -s -D - -H "Origin: http://evil.com" http://localhost:8787/api/collections 2>&1 | grep -i "access-control-allow-origin"
    ```
    Expected: First returns `Access-Control-Allow-Origin: http://localhost:4321`, second returns nothing or is absent.

    SEC-06 — CSRF:
    ```bash
    # Check that CSRF cookie is set on admin page
    curl -s -D - http://localhost:8787/admin 2>&1 | grep -i "csrf_token"
    ```
    Expected: A `Set-Cookie: csrf_token=...` header is present.

    Record ALL results. If ANY check fails, investigate and document what's wrong. Do NOT skip failed checks or fall back to code-level verification.
