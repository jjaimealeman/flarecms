<objective>
Add Cache API to media serving and wire KV namespace into the three-tier cache.

Purpose: Media files are currently fetched from R2 on every request with no edge caching. Content API routes already call `getCacheService()` but KV is never initialized, so all caching falls back to memory-only mode. This plan adds Workers Cache API to the `/files/*` route for edge caching of media, and calls `setGlobalKVNamespace()` in middleware to activate KV as Tier 2 for content API responses.

Output: `/files/*` responses include `Cache-Control: public, max-age=31536000, immutable` and ETag headers with Cache API integration. Content API responses use KV cache (visible in `X-Cache-Status` / `X-Cache-Source` headers).
</objective>

Replace the existing `/files/*` handler in app.ts (currently at ~line 271-307) with a Cache API-enhanced version.

The current handler fetches from R2 every time. The new handler should:

1. Create a cache key from the request URL
2. Check `caches.default` for a cached response — return immediately on hit
3. On miss, fetch from R2 as before
4. Set response headers:
   - Use `object.writeHttpMetadata(headers)` to set Content-Type and Content-Disposition from R2 metadata
   - `etag` set to `object.httpEtag` (RFC 9110 quoted ETag from R2)
   - `Cache-Control: public, max-age=31536000, s-maxage=31536000, immutable`
   - `Access-Control-Allow-Origin: *`
   - `Access-Control-Allow-Methods: GET, HEAD, OPTIONS`
5. Store the response in Cache API via `c.executionCtx.waitUntil(cache.put(cacheKey, response.clone()))` — non-blocking
6. Return the response

**Use `object.writeHttpMetadata(headers)`** instead of manually reading `object.httpMetadata?.contentType` — this is the R2 recommended approach and handles all HTTP metadata fields.

**Fallback for executionCtx:** If `c.executionCtx` is not available (unlikely in Workers, but defensive), skip the cache.put and just return the response.

**Important constraints:**
- Cache API only works on custom domains, not `*.workers.dev` or local dev. `cache.match()` will always miss locally — this is expected, not a bug.
- Do NOT add `X-Cache-Status` headers to the media route — Cache API is transparent and operates at the edge layer.

Replace the full handler block. Keep the try/catch structure. The handler is inside `createSonicJSApp()`.

  <action>
Add a middleware in `createSonicJSApp()` that calls `setGlobalKVNamespace(c.env.CACHE_KV)` before any route handler runs. This activates KV as Tier 2 in the existing `getCacheService()` singleton.

1. Add import at top of app.ts:
```typescript
import { setGlobalKVNamespace } from './plugins/cache'
```

2. Add a module-level flag outside `createSonicJSApp()`:
```typescript
let kvInitialized = false
```

3. Add middleware BEFORE route registration (before `app.route('/api/content', ...)`), after the existing `app.use('*', ...)` middleware chain:
```typescript
// Wire KV namespace into three-tier cache (must run before any route handler calls getCacheService)
app.use('*', async (c, next) => {
  if (!kvInitialized && c.env.CACHE_KV) {
    setGlobalKVNamespace(c.env.CACHE_KV)
    kvInitialized = true
  }
  await next()
})
```

**Why module-level flag:** Workers reuse the same isolate across requests within a cold start window. `setGlobalKVNamespace` sets a module-level variable, so we only need to call it once. The flag avoids redundant calls on every request.

**Why before routes:** `getCacheService()` creates a singleton on first call. If a route handler calls `getCacheService()` before `setGlobalKVNamespace()` runs, the singleton is created without KV and subsequent `setGlobalKVNamespace` calls won't affect it. The middleware MUST precede all route registrations.

**Do NOT modify the cache plugin code itself** — `setGlobalKVNamespace`, `getCacheService`, and the three-tier logic are already correctly implemented. We are just providing the missing KV namespace reference.
